1. Xano backend

Three steps: tables → paged list endpoint → auth. The endpoint’s response shape is the one hard contract wf-xano depends on.

X1 · Database tables

Creates an example opportunities + brands pair — swap in your own domain. The table reference matters: list responses later expose the joined record as brand.company_name for dot-path binds.

Tokens: {{TABLE}}

AI prompt 
Set up the Xano database tables behind a Webflow list powered by the wf-xano library.

Create a table named `brands` with:
- company_name (text)
(id and created_at are added by Xano automatically.)

Create a table named `{{TABLE}}` with:
- title (text)
- description (text)
- status (enum: Active, Closed, Draft — default Draft)
- project_type (text)
- budget (integer)
- budget_frequency (text — e.g. "fixed", "monthly")
- published_at (timestamp, nullable)
- brand_id (table reference → brands)

Seed sample data: 3 rows in `brands`, and 6+ rows in `{{TABLE}}` with a mix of
statuses and budgets, each referencing a brand via brand_id — enough for a paged
list to have something to render.

Important: keep brand_id as a real table reference. The list endpoint (next step)
resolves it with an addon so each item exposes the joined record as
`brand.company_name` — the Webflow side binds dot paths like
wf-xano-bind="brand.company_name".
Manual checklist 
X1 — Create the tables in the Xano UI

1. Database → Add Table → name it "brands".
2. Add field: company_name (Text). (id and created_at are created automatically.)
3. Database → Add Table → name it "{{TABLE}}".
4. Add fields:
   - title — Text
   - description — Text
   - status — Enum, values: Active, Closed, Draft (default Draft)
   - project_type — Text
   - budget — Integer
   - budget_frequency — Text
   - published_at — Timestamp (allow null)
   - brand_id — Table Reference → brands
5. Open each table's records view and add sample rows: 3 brands, 6+ {{TABLE}}
   rows with mixed statuses, each pointing at a brand via brand_id.

X2 · Paged list endpoint

The heart of the contract: a POST endpoint returning Xano’s standard paging metadata — items / itemsTotal / curPage / pageTotal / nextPage — which wf-xano normalizes. Filter inputs here (status, q, sort) match the Webflow controls in W2 exactly.

Tokens: {{API_GROUP}} {{ENDPOINT_PATH}} {{TABLE}} {{XANO_BASE}}

AI prompt 
Create a Xano API endpoint that serves a paged list to the wf-xano Webflow
library. The response shape is a hard contract — do not rename any keys.

Endpoint: POST /{{ENDPOINT_PATH}}   (API group: {{API_GROUP}})

Inputs:
- page (integer, default 1)
- per_page (integer, default 20)
- status (text, optional)
- q (text, optional — free-text search)
- sort (text, optional — whitelist: "newest", "oldest", "budget_high", "budget_low")

Function stack:
1. Query All Records on the `{{TABLE}}` table with paging ENABLED:
   - Bind "page" to the page input and "per page" to the per_page input.
   - Enable the paging METADATA output (Xano's "include paging metadata"
     option) so the result carries itemsTotal / curPage / pageTotal / nextPage.
   - Add an addon that attaches the referenced `brands` record to each item
     under the output name `brand`, so items expose brand.company_name.
2. Filters — apply each only when its input is non-empty:
   - status: where status == input status (skip when empty or "*")
   - q: where title OR description contains the text, case-insensitive
   - sort: "newest" → published_at desc, "oldest" → published_at asc,
     "budget_high" → budget desc, "budget_low" → budget asc.
     Default published_at desc. Reject values outside the whitelist.
3. Return the paged query result directly.

Response contract (exactly this shape — wf-xano normalizes these keys):
{
  "items": [
    { "id": 1, "title": "…", "description": "…", "status": "Active",
      "project_type": "…", "budget": 500, "budget_frequency": "fixed",
      "published_at": 1719964800000,
      "brand": { "company_name": "…" } }
  ],
  "itemsTotal": 45,
  "curPage": 1,
  "pageTotal": 3,
  "nextPage": 2
}

Notes:
- wf-xano sends POST with a JSON body { ...filters, page, per_page }. If the
  Webflow side is later switched to wf-xano-method="GET", the same values
  arrive as query-string parameters — Xano inputs accept both, so use inputs
  rather than parsing the body manually.
- Leave the endpoint public for now; auth is a separate step.
Manual checklist 
X2 — Build the paged list endpoint in the Xano UI

1. Open (or create) the API group {{API_GROUP}} → Add API Endpoint →
   verb POST, path /{{ENDPOINT_PATH}}.
2. Add inputs: page (Integer, default 1), per_page (Integer, default 20),
   status (Text, optional), q (Text, optional), sort (Text, optional).
3. Add a "Query All Records" function on {{TABLE}}:
   - Paging tab: enable paging; bind page → input page, per page → input
     per_page; tick the option that includes paging metadata in the output.
   - Addons: attach the brands record referenced by brand_id, output name
     "brand".
   - Filters: status == input.status only when input.status is set (and not
     "*"); a case-insensitive contains on title/description from input.q;
     sorting driven by input.sort (newest/oldest/budget_high/budget_low),
     default published_at desc.
4. Set the endpoint response to the query result as-is — the output must be
   the object with items / itemsTotal / curPage / pageTotal / nextPage keys
   (Xano's standard paged-metadata shape; do not restructure it).
5. Publish, then smoke-test from a terminal:

   curl -s -X POST '{{XANO_BASE}}/api:{{API_GROUP}}/{{ENDPOINT_PATH}}' \
     -H 'Content-Type: application/json' \
     -d '{"page":1,"per_page":5}'

   Confirm the response has all five keys and items is an array.

X3 · Auth — trade-token endpoint + member scoping

wf-xano’s default auth (wf-xano-auth="memberstack") trades the visitor’s Memberstack JWT for a Xano token at {XANO_BASE}/api:{AUTH_GROUP}/auth/trade-token/v3, then sends Authorization: Bearer on every list request. Rows are scoped server-side from $auth — never by a client-sent id.

Tokens: {{AUTH_GROUP}} {{API_GROUP}} {{ENDPOINT_PATH}} {{XANO_BASE}}

AI prompt 
Add Memberstack authentication to a Xano backend used by the wf-xano Webflow
library.

Part A — trade-token endpoint
Create: POST /auth/trade-token/v3 in API group {{AUTH_GROUP}}.
(wf-xano's default trade URL is
{{XANO_BASE}}/api:{{AUTH_GROUP}}/auth/trade-token/v3 — keep this path unless
WfXanoConfig.tradeTokenPath is also changed on the Webflow side.)

- Input: token (text) — a Memberstack member JWT from the POST JSON body.
- Function stack:
  1. Verify the JWT against Memberstack (Memberstack's verify endpoint or
     JWT decode with the app's public key) and extract member id + email.
  2. Find the row in the `users` table where memberstack_id == member id;
     create it if missing (store memberstack_id and email). The users table
     must be configured as the workspace's authentication table.
  3. Create a Xano authentication token ("Create Authentication Token"
     function) for that user record with a sensible TTL (e.g. 86400).
  4. Return the token. wf-xano accepts either the raw token string as the
     whole response body, or a JSON object { "authToken": "…" }
     ({ "token": "…" } also works).

Part B — lock down and member-scope the list endpoint
- Mark the list endpoint ({{API_GROUP}}: POST /{{ENDPOINT_PATH}}) — or its
  whole API group — as requiring authentication, so requests without a valid
  Bearer token get a 401.
- In its Query All Records, add a where clause scoping rows to the
  authenticated user: e.g. brand_id == the brand owned by the user record
  behind $auth (the token's user id). Scoping must derive from $auth
  server-side — never trust a client-sent member id.
Manual checklist 
X3 — Wire up Memberstack auth in the Xano UI

1. Create a `users` table (if missing): memberstack_id (Text, unique),
   email (Text). Enable it as the authentication table (workspace settings →
   Authentication).
2. In API group {{AUTH_GROUP}}, add endpoint POST /auth/trade-token/v3 with
   input token (Text).
3. Function stack: verify the Memberstack JWT (External API Request to
   Memberstack), find-or-create the users row by memberstack_id, then
   "Create Authentication Token" for that row (TTL e.g. 86400).
4. Return the token — raw string or { "authToken": token }.
5. Open the list endpoint from X2 → require authentication on it (or its
   API group).
6. In its Query All Records, add a where clause tying rows to $auth (e.g.
   via the user's brand), so each member only sees their own rows.
7. Verify: logged out, the list endpoint returns 401. On the published
   Webflow page (memberstack-x loaded first), wf-xano trades the JWT
   automatically — wf-xano-auth="memberstack" is the default — and the
   list renders member-scoped rows.

2. Webflow markup

Each structure ships three ways: an HTML snippet for an Embed element (guaranteed path), an AI prompt for building native Designer elements, and a Copy for Webflow button experimental that puts native elements on your clipboard — click it, then paste straight onto the Designer canvas.

W1 · Basic list structure

Wrapper + template card (binds, date format, status pill, whole-card link) + empty/loader/error states. The minimum viable wf-xano list.

Tokens: {{API_GROUP}} {{ENDPOINT_PATH}} {{DETAIL_URL_PREFIX}}

HTML snippet (paste into a Webflow Embed) 
<div wf-xano-element="wrapper" wf-xano-source="{{API_GROUP}}:{{ENDPOINT_PATH}}">
  <div wf-xano-element="list">
    <a wf-xano-element="template" wf-xano-link="id" wf-xano-link-prefix="{{DETAIL_URL_PREFIX}}">
      <h3 wf-xano-bind="title"></h3>
      <p wf-xano-bind="description"></p>
      <div>
        <span wf-xano-bind="brand.company_name"></span> ·
        <time wf-xano-bind="published_at" wf-xano-format="date"></time>
      </div>
      <span wf-xano-if="status === 'Active'" wf-xano-display="inline-block">Active</span>
    </a>
  </div>
  <div wf-xano-element="empty" style="display:none">Nothing here yet.</div>
  <div wf-xano-element="loader">Loading…</div>
  <div wf-xano-element="error" style="display:none">Something went wrong — try reloading.</div>
</div>
AI prompt (native Designer elements) 
Build a Webflow structure for a wf-xano list. Create NATIVE Webflow elements
(not an Embed) and set each custom attribute exactly as written — every
attribute needs a value, because Webflow strips valueless custom attributes.

1. Div Block — class "opps-wrapper"
   - wf-xano-element = wrapper
   - wf-xano-source = {{API_GROUP}}:{{ENDPOINT_PATH}}
2. Inside it: Div Block — class "opps-list"
   - wf-xano-element = list
3. Inside the list: Link Block — class "opps-card"
   - wf-xano-element = template
   - wf-xano-link = id
   - wf-xano-link-prefix = {{DETAIL_URL_PREFIX}}
   Inside the card:
   - Heading H3 — class "opps-card-title" — wf-xano-bind = title
   - Paragraph — class "opps-card-desc" — wf-xano-bind = description
   - Text Block — class "opps-card-brand" — wf-xano-bind = brand.company_name
   - Text Block — class "opps-card-date" —
     wf-xano-bind = published_at, wf-xano-format = date
   - Text Block — class "opps-card-badge" — text "Active" —
     wf-xano-if = status === 'Active', wf-xano-display = inline-block
4. Back inside the wrapper, as siblings of the list:
   - Div Block "opps-empty" — wf-xano-element = empty — text "Nothing here yet."
   - Div Block "opps-loader" — wf-xano-element = loader — text "Loading…"
   - Div Block "opps-error" — wf-xano-element = error — text "Something went wrong."

The template card is cloned once per item automatically — do not add more
cards. The empty/error elements start hidden (display: none); wf-xano shows
them when needed.
What "Copy for Webflow" pastes
Click the button, focus the Designer canvas where the list should go, and press Cmd/Ctrl+V. Native elements arrive with classes and all wf-xano-* attributes set. The clipboard format is unofficial — if nothing pastes, use the HTML snippet in an Embed instead.

AI prompt (native Designer elements) 
Build a full Webflow list UI for wf-xano: filters, search, sort, counts,
filter tags, clear, and pagination. Create NATIVE Webflow elements and set
each custom attribute exactly as written — every attribute needs a value
(Webflow strips valueless custom attributes). All controls carry
wf-xano-instance = opps so they can live anywhere on the page.

A. Controls (Div Block "opps-controls", anywhere on the page):
   - Text input — wf-xano-search = q, wf-xano-instance = opps
   - Select — wf-xano-filter = status, wf-xano-instance = opps — options:
     "All statuses" with value * (the match-all sentinel: the Designer
     cannot author an empty value, so wf-xano treats * as "clear this
     field"), then Active, Closed, Draft.
   - Select — wf-xano-sort = sort (the value names the request param; "sort"
     is also the default), wf-xano-instance = opps — options newest / oldest /
     budget_high / budget_low (these exact values; the endpoint whitelists them).
   - Button — wf-xano-element = clear, wf-xano-instance = opps — text
     "Clear filters".

B. Counts (Paragraph, anywhere): three inline Text Blocks with
   wf-xano-element = count-from / count-to / total, each with
   wf-xano-instance = opps, arranged as "Showing X–Y of Z".

C. Filter tags (Div Block "opps-tags", anywhere): one Span —
   wf-xano-element = tag, wf-xano-instance = opps, display none — containing
   Text Blocks with wf-xano-element = tag-field and tag-value, and a Button
   with wf-xano-element = tag-remove (text "×"). It is cloned per active
   filter value automatically.

D. The list (Div Block "opps-wrapper"):
   - wf-xano-element = wrapper
   - wf-xano-instance = opps
   - wf-xano-source = {{API_GROUP}}:{{ENDPOINT_PATH}}
   - wf-xano-per-page = 12
   - wf-xano-url-sync = true
   Inside, the same structure as the basic list: a Div Block "opps-list"
   (wf-xano-element = list) holding one Link Block card
   (wf-xano-element = template, wf-xano-link = id,
   wf-xano-link-prefix = {{DETAIL_URL_PREFIX}}) with heading/paragraph/text
   binds (title, description, brand.company_name, project_type, budget,
   published_at with wf-xano-format = date) and an "Active" badge
   (wf-xano-if = status === 'Active'); then sibling Div Blocks for
   wf-xano-element = empty / loader / error (empty and error display: none).
   - Pagination (Div Block "opps-pagination" inside the wrapper): three

3. Recipes

End-to-end paths through the entries above.

R1 · Member-scoped job board

The full stack: tables → paged endpoint → Memberstack auth → install snippet → full list UI. The mega-prompt bundles all three Xano prompts with a contract preamble for a single AI session.

Tokens: {{TABLE}} {{API_GROUP}} {{ENDPOINT_PATH}} {{AUTH_GROUP}} {{XANO_BASE}}

Mega-prompt preamble (prepended to X1–X3 on copy) 
You are setting up a complete Xano backend for a member-scoped list rendered
in Webflow by the wf-xano library (https://github.com/the-starters/wf-xano).
Work through the three steps below in order: tables, then the paged list
endpoint, then Memberstack auth. Two things are hard contracts:
(1) the list endpoint's response keys — items / itemsTotal / curPage /
    pageTotal / nextPage — must not be renamed;
(2) the trade-token endpoint's path — /auth/trade-token/v3 — must not change.
Wiring checklist 
R1 — Wire the halves together

1. Xano: run the three backend steps in order — X1 tables, X2 list endpoint,
   X3 auth (or paste the mega-prompt into one AI session).
2. Webflow → page settings → custom code (before </body>): paste the W3
   install snippet AFTER the memberstack-x script tag.
3. Webflow page: add the W2 structure (Embed snippet, AI prompt, or the
   Copy-for-Webflow paste).
4. Contract checkpoints:
   - the wrapper's wf-xano-source ({{API_GROUP}}:{{ENDPOINT_PATH}}) names the
     endpoint created in X2
   - the endpoint responds with exactly items / itemsTotal / curPage /
     pageTotal / nextPage
   - control field names (status, q, sort) match the endpoint inputs
   - wf-xano-auth defaults to memberstack — memberstack-x must load first,
     and the trade-token endpoint from X3 must be live at
     {{XANO_BASE}}/api:{{AUTH_GROUP}}/auth/trade-token/v3
   - wf-xano-link-prefix points at a real detail page that reads the id
5. Publish and open the page logged in as a member who owns rows: the list
   should render member-scoped data with working filters, search, sort,
   pagination, and URL sync (reload keeps page + filters).

R2 · Public read-only list

The no-auth variant — a public endpoint and the same markup.

Tokens: {{API_GROUP}} {{ENDPOINT_PATH}}

Checklist 
R2 — Public read-only list (differences from the job board recipe)

1. Skip the auth step (X3) entirely; leave the X2 endpoint public.
2. On the wrapper add wf-xano-auth="none". Optionally add
   wf-xano-method="GET" to send params as a query string — the X2 endpoint's
   inputs accept both POST bodies and query params.
3. In the W3 install snippet, add preAuth: false to WfXanoConfig — no list
   needs a token, so skip the parse-time handshake:
   window.WfXanoConfig = { xanoBase: '{{XANO_BASE}}', preAuth: false }
4. Everything else is identical: same endpoint contract, same W1/W2 markup.
